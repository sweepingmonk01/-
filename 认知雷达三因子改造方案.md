# 认知雷达三因子改造方案

## 1. 结论

这次改造不应该做成 `focus / frustration / joy -> 时间 / 信噪比 / 情绪` 的机械换名，而应该把“AI Active 原生内核”和“前端仪表盘投影”拆开。

推荐目标：

1. 后端以 `AIActiveKernelState` 作为唯一认知内核真源，三轴固定为 `time`、`signalNoiseRatio`、`emotion`。
2. `confidence`、`fatigue` 不再和雷达三轴并列为产品主轴，而是降级为编排辅助指标或派生执行态。
3. 前端 dashboard、overview、E.M.A 陪伴态都消费“投影态”，不再直接绑定后端内部字段。
4. 全链路通过兼容归一层同时接受旧结构和新结构，避免历史 JSON snapshot、profile、前端旧字段直接失效。

## 2. 现状问题

当前 `focus / frustration / joy` 已经穿透多层：

- 后端核心类型写死在 `CognitiveState`
- 策略调度直接消费 `frustration`
- 剧情情绪选择直接消费 `focus`
- student profile 和 snapshot 把旧字段作为长期存储
- dashboard / overview / theater / companion sprite 都直接读旧字段

关键位置：

- `server/src/modules/mobius/domain/types.ts`
- `server/src/modules/mobius/infrastructure/heuristic-cognitive-engine.ts`
- `server/src/modules/mobius/infrastructure/strategy-policy-features.ts`
- `server/src/modules/mobius/infrastructure/heuristic-story-planner.ts`
- `server/src/modules/student-state/application/state-update-engine.ts`
- `server/src/modules/student-state/application/student-state-summary-service.ts`
- `server/src/modules/student-state/presentation/student-profile-controller.ts`
- `shared/dashboard-metrics.ts`
- `shared/overview-metrics.ts`
- `src/App.tsx`
- `src/components/Dashboard.tsx`
- `src/lib/firebase.ts`
- `src/lib/mobius.ts`

这意味着如果只改 dashboard 标签，会出现三个问题：

1. 展示名改了，但后端仍然按旧心理学轴做决策，产品内核名实不符。
2. 历史 profile / snapshot 与新前端字段不兼容。
3. 测试、策略和文案会同时失真。

## 3. 目标模型

### 3.1 推荐的领域模型

```ts
export interface AIActiveKernelState {
  time: number;
  signalNoiseRatio: number;
  emotion: number;
}

export interface CognitiveExecutionState {
  confidence: number;
  fatigue: number;
}

export interface CognitiveState {
  schemaVersion: 'ai-active-v2';
  kernel: AIActiveKernelState;
  execution: CognitiveExecutionState;
}
```

说明：

- `time` 不是“已经花掉的分钟数”，而是“时间掌控度 / 节奏余量”，统一用 `0-100` 表达，值越高越稳。
- `signalNoiseRatio` 是“规则信号清晰度”，高值表示题眼、规则、动作链更清晰，噪声更低。
- `emotion` 是“可持续作战情绪”，高值表示更稳、更愿意继续推进。
- `confidence`、`fatigue` 仍保留，但退出雷达三轴，作为调度和判定辅助态。

### 3.2 为什么不建议继续平铺五个字段

如果直接改成：

```ts
{ time, signalNoiseRatio, emotion, confidence, fatigue }
```

虽然改动少，但会继续把“原生内核”和“辅助执行态”混在同一层。后续任何 UI 或业务模块都更容易再次直接耦合到底层字段。

因此建议把三轴内核放到 `kernel` 命名空间，前端再通过投影层消费。

## 4. 旧字段到新内核的语义迁移

### 4.1 旧字段并非一一改名

不建议：

- `focus -> 时间`
- `frustration -> 信噪比`
- `joy -> 情绪`

因为这只是字面替换，语义不成立。

推荐迁移：

- `time`
  - 主要由 `responseTimeMs`、`attempts`、`scrollBurstCount`、`fatigue`、`timeSavedMinutes` 共同决定
  - 表示当前是否还在掌控节奏，而不是单纯“做得快”
- `signalNoiseRatio`
  - 主要由 `wrongStreak`、`correctStreak`、`scrollBurstCount`、近期重复痛点、知识动作命中情况共同决定
  - 表示规则信号是否清晰
- `emotion`
  - 主要由原 `frustration`、`joy`、`confidence` 平滑融合
  - 表示是否还能持续进入下一步动作

### 4.2 兼容映射建议

增加一层归一函数：

```ts
normalizeCognitiveState(input): CognitiveState
```

行为：

1. 如果读到的是新结构，直接返回。
2. 如果读到的是旧结构 `{ focus, frustration, joy, confidence, fatigue }`，映射到新结构。
3. 如果字段缺失，补默认值。

建议映射初版：

```ts
time = clamp(0.45 * focus + 0.25 * (100 - fatigue) + 0.30 * joy)
signalNoiseRatio = clamp(0.55 * focus + 0.25 * confidence + 0.20 * (100 - frustration))
emotion = clamp(0.50 * joy + 0.30 * confidence + 0.20 * (100 - frustration))
```

这层映射只用于历史兼容，不作为长期主计算公式。

## 5. 后端改造方案

### 5.1 类型层

改造文件：

- `server/src/modules/mobius/domain/types.ts`
- `server/src/modules/student-state/domain/types.ts`
- `src/lib/mobius.ts`

动作：

1. 定义 `AIActiveKernelState` 与新的 `CognitiveState`
2. 增加 `schemaVersion`
3. 在前后端共享 DTO 中统一新结构
4. 增加旧结构兼容类型，例如：

```ts
interface LegacyCognitiveState {
  focus?: number;
  frustration?: number;
  joy?: number;
  confidence?: number;
  fatigue?: number;
}
```

### 5.2 归一层

新增建议文件：

- `server/src/modules/mobius/domain/cognitive-state-normalizer.ts`
- `src/lib/cognitive-state.ts`

职责：

1. `normalizeCognitiveState`
2. `createDefaultCognitiveState`
3. `projectDashboardRadarState`
4. `projectLegacyCognitiveState`，仅用于过渡期兼容

### 5.3 认知引擎

改造文件：

- `server/src/modules/mobius/infrastructure/heuristic-cognitive-engine.ts`

现状问题：

- 当前引擎直接输出 `focus / frustration / joy`，本质上是旧世界观。

建议公式方向：

```ts
time = base.time
  - responseTimePenalty
  - extraAttemptPenalty
  - scrollNoisePenalty
  + timeSavedRecovery
  + correctRhythmBonus

signalNoiseRatio = base.signalNoiseRatio
  + correctStreakBonus
  - wrongStreakPenalty
  - scrollNoisePenalty
  - guessLikeAttemptPenalty

emotion = base.emotion
  + correctStreakBonus
  + timeSavedRecovery
  - wrongStreakPenalty
  - repeatedFailurePenalty
```

建议保留：

- `confidence`
- `fatigue`

但它们应写进 `execution`，不再作为雷达三轴。

### 5.4 交互状态迁移引擎

改造文件：

- `server/src/modules/student-state/application/state-update-engine.ts`

现状问题：

- 成功/失败增量仍旧基于旧三轴。

建议改成：

- success
  - `time +6`
  - `signalNoiseRatio +10`
  - `emotion +8`
  - `confidence +12`
  - `fatigue -2`
- failure
  - `time -7`
  - `signalNoiseRatio -12`
  - `emotion -9`
  - `confidence -10`
  - `fatigue +5`

原因：

- 失败首先打击的是“节奏主导权”和“规则清晰度”，而不是单独制造一个叫 `frustration` 的平行轴。

### 5.5 调度策略

改造文件：

- `server/src/modules/mobius/domain/types.ts`
- `server/src/modules/mobius/infrastructure/strategy-policy-features.ts`
- `server/src/modules/mobius/infrastructure/scored-strategy-scheduler.ts`

当前 `frustrationPressure` 必须废弃，替换为三类新特征：

1. `timePressure`
   - 来自 `100 - kernel.time`
   - 时间掌控差时不适合继续扩张任务面
2. `noisePressure`
   - 来自 `100 - kernel.signalNoiseRatio`
   - 信噪比低时应优先缩窄规则和动作
3. `emotionRisk`
   - 来自 `100 - kernel.emotion`
   - 情绪低时优先保护和稳定节奏

推荐策略权重：

- `probe`
  - 适合 `noisePressure` 中高，但 `timePressure` / `emotionRisk` 未崩
- `teach`
  - 对 `timePressure`、`emotionRisk`、近期失败最敏感
- `review`
  - 依赖高 `signalNoiseRatio`、高 `emotion`、近期成功恢复

也就是：

- `teach` 不再由“高挫败”单点触发，而是由“时间吃紧 + 噪声高 + 情绪掉落”的组合触发
- 这更符合 AI Active 的编排逻辑

### 5.6 剧情与提示词

改造文件：

- `server/src/modules/mobius/infrastructure/heuristic-story-planner.ts`
- `server/src/modules/mobius/application/mobius-orchestrator.ts`

建议：

1. 剧情 emotion label 仍可保留 `protective / encouraging / focused / urgent`
2. 但选择依据不再看 `focus > 75`
3. 依据改为：
   - `emotion` 低 -> `protective`
   - `signalNoiseRatio` 高 -> `focused`
   - `review` -> `encouraging`
   - `time` 低或多因素失稳 -> `urgent`
4. 视频 prompt 中把旧字段文案改为新内核文案

示例：

```txt
Student AI Active kernel: time=..., signal-noise-ratio=..., emotion=...
```

### 5.7 Student State / Profile / API

改造文件：

- `server/src/modules/student-state/application/student-state-summary-service.ts`
- `server/src/modules/student-state/presentation/student-profile-controller.ts`
- `server/src/modules/student-state/infrastructure/sqlite-student-profile-repository.ts`
- `server/src/modules/mobius/presentation/mobius-controller.ts`

结论：

- snapshot 与 profile 都存 JSON blob，不需要重 SQL migration
- 但必须做“读取兼容 + 写入新结构”

建议：

1. 写入时统一写新结构
2. 读取时统一走 `normalizeCognitiveState`
3. `recommendedSessionDefaults.previousState` 返回新结构
4. `dashboard-stats` 不再直接透传后端内核字段，而是返回 `dashboard view state`

过渡期建议：

- `createSession`
- `student-state-summary`
- `profile`

这三个接口都同时支持读取旧 `previousState`

## 7. 兼容与迁移策略

### 7.1 为什么不做一次性硬切

当前历史数据源至少有三类：

1. student profile 的 `cognitiveState`
2. student state snapshot 的 `cognitiveState`
3. 前端本地传给 `createSession.previousState` 的旧字段

一次性硬切会导致：

- 老数据读取失败
- 新用户和老用户表现不一致
- 线上接口可能收到旧字段直接报错

### 7.2 推荐迁移策略

Phase 1：

- 加入 normalizer
- 服务端读旧写新
- 前端读新，也兼容旧

Phase 2：

- 前端全面改成新字段
- dashboard / overview / theater 全部切到新投影

Phase 3：

- 测试稳定后移除 legacy 投影和旧字段依赖

### 7.3 数据库层

由于当前 SQLite / file repository 都是 JSON blob：

- 不需要 `ALTER TABLE`
- 也不需要离线批量迁移
- 只需要“读时归一、写时升级”

这是本次改造可控的核心原因。

必须覆盖：

1. 旧 snapshot 能否被新服务正确读取
2. 旧 profile 能否被新前端正确展示
3. `createSession.previousState` 传旧结构时是否仍可创建 session
4. 新结构是否驱动 `teach / probe / review` 发生合理分流
5. dashboard 雷达是否显示 `时间 / 信噪比 / 情绪`
6. overview 是否不再依赖 `frustration`
7. companion / theater 是否不再出现 `focus / frustration / joy` 文案

## 10. 风险与规避

### 风险 1：`时间` 语义不清

规避：

- 明确把它定义成“时间掌控度”，不是原始耗时
- UI 文案可以显示“时间”标签，但 tooltip / 说明里写“节奏余量”

### 风险 2：`timeSavedMinutes` 与 `kernel.time` 重复计分

规避：

- dashboard 估分允许使用 `kernel.time`
- overview 评分避免再次重度依赖 `kernel.time`

### 风险 3：历史数据混杂新旧结构

规避：

- 所有读取入口统一走 normalizer
- 所有写入入口统一落新结构

### 风险 4：测试全线爆红

规避：

- 先上兼容层，再切计算公式
- 先修类型和转换测试，再修策略断言

## 11. 最终建议

如果这次改造的目标真的是把“时间 / 信噪比 / 情绪”升级为产品独有的 AI Active 原生内核，那么最对的做法是：

1. 后端先建立新的 canonical kernel schema
2. 通过 normalizer 兼容旧数据
3. 把策略调度从“挫败驱动”改成“时间-信号-情绪联合驱动”
4. 前端只消费投影，不再直连后端内部字段

这样改完之后，雷达不只是换了三个词，而是整个编排、状态闭环、首页解释、剧场反馈都会真正建立在新的产品内核上。
