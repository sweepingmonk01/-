# 当下优先工程任务拆解

更新时间：2026-04-30

这份文档不讨论愿景，只讨论“现在最该做什么、做到什么算完成、应该动哪些文件”。

## 一句话结论

当前最优先不是继续加新概念，而是先把现有系统补成一个可追踪、可评估、可演进的闭环。

工程顺序应固定为：

1. `P0` 标准化“假设-干预-效果”数据管道
2. `P0` 把学生状态升级成可计算状态向量
3. `P0` 收口 PR 入库边界，隔离生成物和产品代码
4. `P1` 做机器猜想引擎 `v1`
5. `P1` 把启发式编排器替换成可评分策略调度器
6. `P1` 把知识图谱接入决策链路
7. `P2` 扩展多模态感知入口

如果当前只能开工 3 个任务，只做前 3 个，不要分散。`P0-3` 是 PR 卫生任务，不扩大产品范围，只负责让下一轮审查能清楚地区分源码、文档、配置、生成物。

## 为什么是这个顺序

- 当前仓库已经有状态快照、诊断线程、知识图谱、事件日志、作业策略规划，但这些能力还是分散存在。
- 真正缺的不是“再多一个 AI 能力”，而是让系统能回答三个工程问题：
  - 这次系统基于什么状态做了什么判断？
  - 这次干预到底有没有效果？
  - 下次系统能不能利用上一次结果变得更准？
- 只要这三个问题还不能被稳定回答，后面的群体智慧飞轮、开放 API、跨学科迁移都只是提前叙事。

## P0-1：标准化闭环数据模型与事件采集

### 目标

把一次学习过程从“散落在多个模块里的日志和快照”收拢成统一的闭环记录，使后续评估、调参、回放、训练都有稳定输入。

### 当前现状

- 已有通用事件记录：[server/src/modules/analytics/application/sqlite-event-logger.ts](server/src/modules/analytics/application/sqlite-event-logger.ts)
- 已有状态快照：[server/src/modules/student-state/domain/types.ts](server/src/modules/student-state/domain/types.ts)
- 已有会话编排：[server/src/modules/mobius/application/mobius-orchestrator.ts](server/src/modules/mobius/application/mobius-orchestrator.ts)
- 已有诊断线程：[server/src/modules/ai/application/socratic-diagnostic-service.ts](server/src/modules/ai/application/socratic-diagnostic-service.ts)

问题是：

- `analytics_events` 只有 `name + payload_json`，适合埋点，不适合闭环评估。
- `StudentStateSnapshot` 只覆盖 `session-created / interaction-resolved` 两个阶段，信息不够完整。
- 假设、动作选择理由、干预内容、结果验证没有统一主键串起来。

### 交付物

1. 新增一张“学习闭环主表”
2. 给每次会话/干预生成稳定的 `cycleId`
3. 把状态、假设、动作、结果统一挂在 `cycleId` 下
4. 为后续评估查询预留明确字段，而不是只放 JSON blob

### 建议数据结构

建议新增表：`learning_cycles`

核心字段建议：

- `id`
- `student_id`
- `session_id`
- `media_job_id`
- `source`
- `status`：`started | diagnosed | intervened | validated | closed`
- `pain_point`
- `rule`
- `knowledge_action_id`
- `state_before_json`
- `state_after_json`
- `hypothesis_summary_json`
- `selected_action_json`
- `outcome`
- `effect_score`
- `created_at`
- `updated_at`

建议新增明细表：`learning_cycle_events`

核心字段建议：

- `id`
- `cycle_id`
- `event_type`
- `event_payload_json`
- `created_at`

事件类型建议先固定为：

- `cycle.started`
- `state.snapshot.before`
- `hypothesis.generated`
- `action.selected`
- `intervention.delivered`
- `interaction.resolved`
- `state.snapshot.after`
- `effect.evaluated`
- `cycle.closed`

### 影响文件

优先修改：

- `server/src/modules/analytics/application/sqlite-event-logger.ts`
- `server/src/modules/mobius/application/mobius-orchestrator.ts`
- `server/src/modules/ai/application/socratic-diagnostic-service.ts`
- `server/src/app.ts`

建议新增：

- `server/src/modules/analytics/domain/types.ts`
- `server/src/modules/analytics/infrastructure/sqlite-learning-cycle-repository.ts`
- `server/src/modules/analytics/application/learning-cycle-service.ts`

### 实施步骤

1. 定义 `LearningCycle` / `LearningCycleEvent` 类型
2. 建立 sqlite repository 与建表语句
3. 在 `MobiusOrchestrator.buildSession()` 中创建 `cycle.started`
4. 在交互裁决后写入 `interaction.resolved` 和 `state.snapshot.after`
5. 在 `SocraticDiagnosticService` 里把诊断线程与 `cycleId` 关联
6. 保留 `analytics_events` 作为埋点层，不直接替代闭环层

### 验收标准

- 任意一次学习会话都能查到唯一 `cycleId`
- 能从一个 `cycleId` 回放出“状态 -> 假设/动作 -> 结果 -> 新状态”
- 至少有 1 条集成测试覆盖闭环主链路
- 后续做离线统计时，不需要靠解析非结构化日志拼装

## P0-2：学生状态向量层与增量更新器

### 目标

把现在的“摘要式学生状态”升级成“统一、可决策、可比较、可版本化”的状态向量。

### 当前现状

- 当前状态主要定义在 [server/src/modules/mobius/domain/types.ts](server/src/modules/mobius/domain/types.ts) 的 `CognitiveState`
- 当前快照主要定义在 [server/src/modules/student-state/domain/types.ts](server/src/modules/student-state/domain/types.ts)
- 当前汇总逻辑在 [server/src/modules/student-state/application/student-state-summary-service.ts](server/src/modules/student-state/application/student-state-summary-service.ts)

问题是：

- 状态维度还偏少，且主要服务展示，不足以支持策略选择。
- 没有显式置信度、状态来源、版本号、更新时间语义。
- `StudentStateSummaryService` 更像聚合器，还不是状态更新器。

### 交付物

1. 一个正式的 `StudentStateVector`
2. 一个统一的 `StateUpdateEngine`
3. 一个“状态快照”和“状态摘要”分离后的模型边界
4. 给 planner / diagnostic / graph 都暴露同一份状态对象

### 建议类型结构

建议新增：

```ts
interface StudentStateVector {
  version: string;
  studentId: string;
  computedAt: string;
  cognitive: {
    focus: number;
    frustration: number;
    joy: number;
    confidence: number;
    fatigue: number;
  };
  mastery: Record<string, {
    score: number;
    confidence: number;
    lastEvidenceAt?: string;
  }>;
  errorBeliefs: Record<string, {
    score: number;
    evidenceCount: number;
  }>;
  sessionContext: {
    currentPainPoint?: string;
    currentRule?: string;
    recentFailureCount: number;
    recentSuccessCount: number;
  };
}
```

第一版不要求把 `mastery` 做满，但接口必须预留。

### 影响文件

优先修改：

- `server/src/modules/student-state/domain/types.ts`
- `server/src/modules/student-state/application/student-state-summary-service.ts`
- `server/src/modules/mobius/application/mobius-orchestrator.ts`
- `server/src/modules/ai/application/strategic-planner-service.ts`

建议新增：

- `server/src/modules/student-state/application/state-update-engine.ts`
- `server/src/modules/student-state/application/state-vector-service.ts`

### 实施步骤

1. 定义 `StudentStateVector` 类型
2. 拆分“原始快照”和“计算后状态”
3. 建一个增量更新器，输入为：
   - 上一状态
   - 新交互结果
   - 行为信号
   - 诊断结论
4. 把 `StrategicPlannerService` 改为消费 `StudentStateVector`
5. 把前端摘要接口继续保留，但由新状态向量衍生输出

### 验收标准

- `planner`、`diagnostic`、`orchestrator` 使用的是同一状态模型
- 每次交互后能生成前后两个版本的状态向量
- 状态更新规则集中在一个模块，不再散在多个服务里
- 集成测试能证明一次失败/成功会导致状态向量出现预期变化

## P0-3：PR 入库边界与生成物隔离

### 目标

把当前工作区先按“应入库 / 应忽略 / 需人工确认”收口，避免下一轮 PR 审查把构建产物、运行态数据、视觉 QA 截图和产品源码混在一起。

本节基于 2026-04-30 的 `git status --short` 和 `git status --ignored --short` 梳理。后续 Agent 在开 PR 前必须先复核本节，不要直接 `git add .`。

### 自动检查

- `npm run check:worktree`：汇总当前工作区的 `Should commit / Should ignore / Deferred from feature PR / Unknown` 四类清单。

- 严格模式通过不等于可以 `git add .`；`Deferred from feature PR` 仍应保持未入库，除非另开移动端/文档资产 PR。

- `diff_output.txt`：当前已被历史跟踪，像调试/审查输出；如非正式交付物，应单独提交删除
- `tmp_repo/`：当前已被历史跟踪，且 `app/applet/sync_architecture.ts` 依赖该路径；是否保留为迁移桥接目录需要单独确认
- `app/applet/sync_architecture.ts`、`app/applet/universal_server.ts`、根目录 `sync_architecture.ts`、`universal_server.ts`：若仍服务 AI Studio/小程序同步，保留；否则在专门清理 PR 中处理

1. 先运行 `git status --short`，确认未跟踪项已经落入本节三类之一。
2. 只按清单 staged，不使用 `git add .`。

4. 如需纳入“需人工确认”项，PR 描述必须单独说明原因和审查范围。

## P1-1：机器猜想引擎 v1

### 目标

让系统从“会追问”升级到“会提出候选假设、设计探测动作、根据反馈更新判断”。

### 当前现状

- [server/src/modules/ai/application/socratic-diagnostic-service.ts](server/src/modules/ai/application/socratic-diagnostic-service.ts) 已经能创建诊断线程和连续追问
- [server/src/modules/mobius/application/mobius-orchestrator.ts](server/src/modules/mobius/application/mobius-orchestrator.ts) 已经能根据交互提交做裁决

问题是：

- 还没有显式的“候选假设集”
- 没有“探测动作”与“普通干预动作”的区分
- 没有置信度更新
- 没有把最终诊断结论写回后续动作选择

### 交付物

1. `HypothesisCandidate[]`
2. `ProbeAction`
3. `HypothesisUpdateResult`
4. 一个最小可用的 `HypothesisEngine`

### 建议类型结构

建议新增：

```ts
interface HypothesisCandidate {
  id: string;
  label: string;
  rationale: string;
  confidence: number;
  evidence: string[];
}

interface ProbeAction {
  id: string;
  type: 'question' | 'prompt' | 'micro-task';
  prompt: string;
  targetHypothesisIds: string[];
}
```

### 影响文件

优先修改：

- `server/src/modules/ai/domain/types.ts`
- `server/src/modules/ai/application/socratic-diagnostic-service.ts`
- `server/src/modules/ai/presentation/ai-controller.ts`

建议新增：

- `server/src/modules/ai/application/hypothesis-engine.ts`
- `server/src/modules/ai/application/hypothesis-engine.test.ts`

### 实施步骤

1. 定义候选假设类型和持久化位置
2. 在失败场景中先生成 2 到 3 个候选假设
3. 为假设生成一个“最小探测动作”
4. 学生响应后更新假设置信度
5. 选出当前最可信假设，产出针对性干预建议
6. 把最终选中的假设写回 `learning_cycles`

### 第一版范围控制

第一版不要做这些：

- 全量贝叶斯模型
- 跨学生假设迁移
- 复杂多轮树搜索
- 自动训练动作策略

第一版只需要做到：

- 能生成候选假设
- 能根据一次追问或一次探测更新排序
- 能把最可信假设用于下一步动作选择

### 验收标准

- 一次失败后，系统能产出结构化候选假设，而不是只有自然语言追问
- 学生回答一次后，系统能更新假设排序
- 最终输出的干预建议与最高置信假设绑定
- 至少有单测覆盖“生成 -> 更新 -> 选择”主链路
