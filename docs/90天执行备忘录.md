# 90 天执行备忘录

更新日期：2026-05-08

本文不讨论愿景，也不重复路线图，只回答一个问题：**未来 90 天内，列子御风必须做完哪几件事，才能让"AI Active 范式"从思想变成事实。**

适用对象：主理人、工程负责人、未来接手 P0/P1 任务的 Agent。

## 一句话总纲

> 范式革命产品在第一年活下来的唯一方法，是用旧范式的语言交付价值，同时用新范式的架构积累壁垒。

列子御风当前架构已经在做后半句，本备忘录解决前半句以及"前后两句怎么不互相吃掉"的工程纪律。

## 一、前 30 天：把"半闭环"补成"真闭环"

### 1.1 learning_cycles 端到端最小可演示链路

不要等表设计完美，先做最小可用版。

**强制纪律**：

- `cycleId` 必须在 `MobiusOrchestrator.buildSession()` 入口处生成，**从此刻起所有日志强制带它**。这是飞轮的"标识符纪律"，不可妥协。
- `state_before_json` / `state_after_json` 第一版直接落现成的 `cognitiveState`，不要等状态向量重构完。
- `effect_score` 第一版就用最朴素的 `(本周期成功率 - 上一周期成功率)`，**先有数字，再优化定义**。

**验收标准**：能在 demo 时打开 SQLite，按一个 `cycleId` 查到一条完整的"状态前 → 假设 → 干预 → 结果 → 状态后"五段式记录。

**反模式**：不允许"等状态向量重构完成再写入 cycle"。这种顺序会让 cycle 永远不存在。

### 1.2 砍掉 src/App.tsx 的 2767 行单点风险

这是当前最紧迫的工程债，会拖慢后续每一次改动。

**动作**：

- 把"今日任务编排"逻辑抽成 `useDailyOrchestration()` hook，单独可测。
- 把 cogState / sessionState / errorBookState 三类状态分进三个 context provider。
- App.tsx 最终只保留 layout 和路由，目标行数 < 400。

**判断标准**：抽取完成后，新加一个首页卡片不需要修改 App.tsx 主体。

### 1.3 暂停所有非闭环新功能入仓

包括但不限于：

- 基础科学双引擎的前端入口
- 生成式漫剧扩展
- 新的可视化面板
- 任何"看起来很酷"的 demo 页

理由：它们全部依赖闭环数据存在，提前做就是在没有地基的地上盖二楼。

[当下优先工程任务拆解.md](../当下优先工程任务拆解.md) 的"不要现在优先做的事"列表必须当作红线执行，不只是建议。

## 二、30-60 天：把"启发式"逼到墙角

启发式本身不是问题，没有把启发式标注为启发式才是。

### 2.1 给每个启发式公式打"可证伪标签"

比如 `state-update-engine` 里 success 的 `time +6, snr +10, emotion +8` 这组数字，代码注释必须显式声明：

```
这组系数是 v0 的工程估值。
将在累计 N=200 cycles 后替换为基于 effect_score 回归的版本。
替换验收：7 天内 cycle 完成率下降不超过 5%。
```

**本质**：用工程承诺把启发式锁死在临时状态，避免它无意识地变成永久结构。

### 2.2 准备策略 A/B 框架

[scored-strategy-scheduler.ts](../server/src/modules/mobius/infrastructure/scored-strategy-scheduler.ts) 当前 89 行是占位，方向对了。60 天内必须做到：

- 同一个状态向量同时跑 `heuristic` 与 `scored` 两条策略
- 选择由配置决定，但两条策略的输出**全部写入 cycle**
- 数千条 cycle 之后即可离线 replay 对比 effect_score 分布

**关键**：这是一个不需要训练、不需要算力的飞轮启动器，只需要工程纪律。

### 2.3 状态向量必须落地，但只做骨架

[当下优先工程任务拆解.md](../当下优先工程任务拆解.md) 的 P0-2 中 `mastery` 字段第一版可以全空，但接口必须预留。**让 planner / diagnostic / orchestrator 真的共用同一个状态对象**比让状态丰富更重要。

## 三、60-90 天：建立第一份"对外说得清"的效果证据

范式革命产品最大的风险不是技术，是叙事腐败——内部觉得很对，对外说不清。

### 3.1 强制产出一份双盲式 mini study

**设计**：

- 招募 30-50 名学生，分 A/B 两组
  - A 组：列子御风 AI Active 路径
  - B 组：同样题库 + 传统"刷题+订正"路径
- 时长：至少 4 周
- 主指标：**同难度新题首做正确率的提升幅度**（不是分数，分数受太多干扰）
- 次指标：每次干预的平均时间消耗

**为什么必须做**：

1. 它是把范式从"思想"变成"事实"的唯一方式
2. 它是融资和拓客时唯一能扛住质询的硬通货
3. 它倒逼工程把 effect_score 真的做对，而不是写成路线图字符串

**反模式**：不允许用 NPS、留存、完课率代替学习效果指标。这些指标在旧范式下定义，无法证明新范式价值。

### 3.2 90 天检查点

如果 90 天后没有以下三件东西同时出现，必须重新评估资源分配：

1. 至少一份完整 cycle 的端到端 SQL 记录
2. 至少一份 mini study 的初步数据（即便样本不足）
3. 至少一次内部演示能连贯说出"状态前 → 干预 → 状态后"

## 四、产品叙事纪律（永久执行）

### 4.1 三套语言，三个对象，永不混用

| 对象 | 用语 | 原因 |
| --- | --- | --- |
| 工程师 / 投资人 | AI Active / 状态驱动 / 元知识底座 | 他们识别架构稀缺性 |
| 家长 / 学生 | "先看见你卡在哪儿，再带你走" | 范式优势的可感知投影 |
| 教师 / 学校 | "每个学生一份可解释的状态报告" | 击中个性化指导的真实痛点 |

[产品技术内核与市场定位分析.md](../产品技术内核与市场定位分析.md) 的"对外定位话术"已经做了这个区分，要把它升级为**永久纪律**，每次版本迭代都回头检查是否走偏。

### 4.2 短期不把 "AI Active" 作为家长侧 slogan

家长不买"主动式学习引擎"，家长买"下周考试能多对几道题"。

技术内核对内坚持，对外翻译成结果语言。这不是妥协，是范式革命产品 12-18 个月内活下来的唯一路径。

## 五、风险登记簿

| 风险 | 等级 | 触发条件 | 缓解动作 |

| learning_cycles 永远停留在路线图 | 高 | 30 天内未跑通端到端链路 | 砍其他 P1 任务，全员压上 |
